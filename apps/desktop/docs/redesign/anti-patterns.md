# Anti-Patterns

## 1. The endless assistant monologue
If progress can only be understood by reading a long assistant response, the interface is failing.

## 2. Thinking without state
A spinner or vague “thinking” label is not enough for an agent system doing meaningful work.

## 3. Hiding actions to feel magical
Opacity may feel sleek in demos, but it erodes trust in real use.

## 4. Treating approvals like modal interruptions only
Approvals should be integrated into the run model, not bolted on as jarring popups.

## 5. Confusing chronology with clarity
A perfect timeline is still a bad UI if users cannot quickly answer “what matters now?”

## 6. Making all tasks look identical
Research, coding, writing, browsing, and messaging tasks may share a run model but need different artifact emphasis.

## 7. Completion without review
“Done” is not enough. The user needs a review surface.

## 8. Recovery that feels like a reset
If interruption or failure forces the user to start over mentally, the system loses credibility.

## 9. Overusing chat as a container
Chat should not be the default vessel for plans, logs, approvals, outputs, and verification all at once.

## 10. Polished confidence without evidence
Confidence should emerge from observable checks, not tone.
