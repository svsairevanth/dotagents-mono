# Product Principles

## 1. Task-first, not transcript-first
The primary object is the task run. Messages support the task; they do not define the product.

## 2. Progress over prose
Show state, actions, outputs, and blockers before long narrative explanations.

## 3. Legibility over magic
The system should feel capable but never opaque. Hidden execution erodes trust.

## 4. Interruptibility everywhere
Any meaningful process should be pausable, cancelable, redirectable, or resumable.

## 5. Review before acceptance
Users need a clear review state before a task is considered complete.

## 6. Progressive autonomy
Higher autonomy should unlock through trust, verification, and user comfort, not be assumed from the start.

## 7. Artifacts matter more than dialogue
Files, diffs, drafts, screenshots, links, and created resources are the real output.

## 8. The user should always know what needs them
Approvals, questions, blockers, and ambiguity should be surfaced explicitly.

## 9. Parallel work should feel structured
When multiple agents or subtasks run at once, the UI should expose orchestration without becoming chaotic.

## 10. Failure should degrade clearly
If a tool fails, context is missing, or confidence drops, the UI should narrow options and surface the issue plainly.

## 11. Default to calm density
The product should convey a lot without feeling cluttered. Dense does not mean noisy.

## 12. Natural language remains important
Conversation should remain available for steering, clarification, and refinement, but it should support the workflow rather than consume it.
