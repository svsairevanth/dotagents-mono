# Product Spec

## Summary

Redesign `.agents` around a task-centric workspace that separates intent, execution, and review. Replace the transcript-first center of gravity with a run-based UI that makes active work, artifacts, approvals, and completion state much easier to understand and control.

## Problem

The current UI appears to inherit chat-product assumptions. That makes it harder to:

- scan progress quickly
- understand current execution state
- review results with confidence
- manage multiple sessions as work objects
- distinguish planning, execution, and completion
- expose approvals and interventions clearly

## Goals

- Make the task / run the primary UI object
- Make active execution glanceable
- Make tool and agent activity legible without transcript digging
- Make outputs easy to review
- Preserve the speed and flexibility of chat input
- Support both single-task focus and multi-run overview
- Increase user trust in autonomous execution

## Non-goals

- Do not remove chat entirely
- Do not expose raw chain-of-thought
- Do not force complex project-management ceremony for lightweight use
- Do not overfit the design to code-only workflows

## Primary user questions

The UI must answer these quickly:

- What is happening right now?
- What has already happened?
- What is the system trying to do next?
- What requires my approval or input?
- What was produced?
- Can I trust the current state?

## Core object model

### Task
A user-defined unit of work with a goal and context.

### Run
A specific execution instance of a task.

### Step
A meaningful stage in a run, such as planning, research, implementation, verification, or handoff.

### Action
An atomic action taken by an agent or tool.

### Artifact
A concrete output such as a diff, file, summary, screenshot, document, issue, or link.

### Review state
The human-facing assessment state of the output, such as needs review, verified, approved, or follow-up requested.

## Main surfaces

### 1. Work Queue / Runs Sidebar
Purpose: provide a scan-friendly list of active, blocked, paused, and completed work.

Should show:

- run title
- status
- active step
- approval needed badge
- freshness / last activity
- priority or pinned state
- completion state

### 2. Task Workspace
Purpose: become the primary working surface for one run.

Default sections:

- Goal
- Plan
- Current step
- Recent actions
- Artifacts
- Risks / blockers
- Verification
- Follow-up input

### 3. Live Activity Stream
Purpose: show time-ordered execution events without forcing transcript reading.

Should include:

- agent actions
- tool invocations
- approvals
- warnings
- external side effects
- handoffs

### 4. Review Surface
Purpose: turn final output into a structured review workflow.

Should support:

- artifact inspection
- file diffs
- command summaries
- source lists
- acceptance / revision controls
- continue-from-here actions

## Main layout

Recommended default layout:

- **Left rail:** work queue and navigation
- **Center:** focused task workspace
- **Right rail:** live activity, approvals, and artifacts

This turns the product into an operations console rather than a transcript viewer.

## Run states

A run should have explicit states, for example:

- Draft
- Queued
- Planning
- Running
- Awaiting approval
- Blocked
- Verifying
- Ready for review
- Completed
- Failed
- Stopped

These should be visually distinct and machine-readable.

## Default task workspace structure

### Goal card
A compact restatement of what the user asked for.

### Plan card
A short, editable plan generated by the system or user.

### Current step card
Shows what is active now, why it matters, and what success for the step looks like.

### Progress card
Summarizes:

- steps complete
- active sub-agents
- tool activity count
- elapsed time
- blockers

### Artifacts panel
Shows structured outputs grouped by type.

### Review panel
Shows verification results, unresolved risks, and acceptance controls.

### Steering input
Persistent follow-up input for redirecting the run without hunting for the transcript.

## Approval model

Approvals should be elevated out of the transcript and shown as explicit cards.

Each approval card should include:

- what action is proposed
- why it is needed
- what could change
- risk level
- approve / deny / edit options

## Multi-run overview

The overview page should feel like a workbench, not a gallery of chats.

Useful sections:

- Active
- Needs input
- Needs review
- Background runs
- Completed recently

The emphasis should be on operational status, not chronology alone.

## Success criteria

A successful redesign would make it possible for a user to open the app and within a few seconds know:

- which work matters right now
- which run is blocked
- what has already been produced
- what needs their attention
- whether the system is safe to leave running
